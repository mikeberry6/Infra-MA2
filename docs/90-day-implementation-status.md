# InfraSight 90-Day Program — Implementation Status

**Updated:** 2026-07-22

**Posture:** stabilized Next 15 repository implementation is substantially complete; isolated validation, production rollout, elapsed telemetry, and the separately gated modernization release remain open.

## Implemented in the repository

- Node 24/npm 11 policy, patched supported framework/auth/Prisma lines, locked npm installation, and a zero-high/critical production dependency gate.
- Fixed seed credentials removed; explicit administrator bootstrap, durable login throttling, callback validation, security headers, and role guards added.
- Additive operational schema foundations (`PipelineRun`, `AuditEvent`, `CompanyRedirect`, `AuthThrottle`, `lastVerifiedAt`) and migration files added.
- Dashboard/news/import/weekly synchronization instrumentation, public freshness states, scheduled workflows, retry bounds, source/provider thresholds, rolling reliability reports, and retained run artifacts added. Failed weekly synchronization records retain accumulated inserted/updated/skipped counts and bounded lifecycle metadata instead of collapsing partial progress to zero. The nightly news job now pins a deterministic 200-entity UTC service-day window (about an eight-day current cycle), derives a cycle-plus-two-day lookback, prioritizes the required official-source window inside its reviewed page budget, and distinguishes intentional deferral from incomplete coverage. Real upstream failures remain fail-closed above 25%; crawler, robots, and news-search requests use per-hop public-address validation, DNS-pinned connections, bounded redirects/responses/timeouts, default web ports, and canonical final-URL attribution. Dashboard coverage counts only fully successful current provider runs and identifies missing or stale required metrics.
- Publication/source gates, reviewer-neutral citation, ownership-link, Fund primary-source, and deal seller-disclosure remediation, audited mutation foundations, reviewed duplicate-merge/redirect tooling, and source-coverage reporting added. Fund publication/import/admin validation now requires a currency-denominated amount or explicit `TBD`; ambiguous placeholders cannot pass. `Fund.primarySourceUrl` is an additive, explicitly reviewed designation; existing supporting links are never auto-promoted. No migration or release step auto-designates a primary citation, Fund primary source, canonical survivor, ownership vehicle, or seller-disclosure treatment.
- Database metrics/rankings, 25-row initial public pages, fixed 25-row URL-addressable admin pagination, shareable URL state, mobile filter sheets, accessible drawers, canonical `/tracker`, globally ranked and entity-scoped grouped search, preview/confirm imports, list/detail payload separation, detail APIs, analytics, and request-correlated structured route/provider timing added. Search pagination preserves database tie ordering so pages remain disjoint and complete, while direct/search/history drawer opens share the same exactly-once analytics and shell measurement as row clicks. Drawer click-to-committed-shell timing is payload-free and has a 100 ms browser regression budget without waiting for lazy detail data.
- The public health endpoint is intentionally minimal: its top-level response contains only `status`, `version`, `generatedAt`, `database`, `pipelines`, and `generationTimeMs`; it does not disclose schema, host, branch, credentials, or query details.
- Reliability artifacts distinguish `collecting`, `healthy`, and `unhealthy`, preserve a separate `operationallyHealthy` signal, and set the program exit criterion only after the complete 30-day observation window passes. Scheduled diagnostics continue during collection, while production promotion uses `--require-full-window` and fails closed until the elapsed-window requirement is met.
- Pipeline workflows preserve useful failure evidence: reliability checks run after a failed scan when installation succeeded, weekly/monthly checks aggregate independent failures instead of stopping at the first one, and artifact uploads run even when an earlier diagnostic fails.
- Unit/integration tests, Playwright anonymous/authentication checks, axe checks, keyboard/focus checks, responsive overflow checks, mobile primary-navigation coverage, seeded external-provider failure journeys, and visual baselines are present. The isolated validation job now runs visual baselines serially before mutation journeys, records the main browser and independent failure-fixture evidence after an editorial-data failure, and enforces all browser, visual, and strict-data outcomes again at the end, so evidence collection cannot weaken release blocking.
- Protected-branch release aggregation, isolated migration/data/browser gate, reviewed data-remediation workflow, schema-first production promotion, rollback workflow, and release/incident/governance documentation are present. The isolated browser gate includes a separate loopback server with a deliberately unreachable database for top-level failure/retry journeys. Production operations and data pipelines share one non-cancelling lock; staging and dashboard cutovers require an explicit dashboard-write freeze. Credentialed on-demand data and LinkedIn collection use default-branch-only dispatch with exact checked-out-SHA verification, and third-party workflow actions are immutable-SHA pinned to current Node 24 runtimes.
- Release provenance is fail-closed in code: the requested SHA must equal the protected `main` head, its `build` check must be a successful GitHub Actions check for that exact SHA, migration manifests hash committed release blobs, and promotion validates the staged production deployment's Vercel project/target/Git SHA before changing domains.
- The application includes Web Analytics and Speed Insights. Vercel is verified to track `main` on Node 24 with automatic production-domain assignment disabled.
